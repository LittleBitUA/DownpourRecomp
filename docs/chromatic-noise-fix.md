# Chromatic-noise fix — depth → 7e3 ownership transfer

## The bug

Silent Hill: Downpour exhibits a UE3 G-buffer EDRAM-aliasing pattern that triggers a depth → 7e3 (`k_2_10_10_10_FLOAT`) ownership transfer in the rexglue HostRenderTargets path. The transfer pixel shader treats the depth bits as packed 7e3 floats and decodes them via `Float7e3To32`, producing values like `(8.1875, 0.25, 0.8125, 0.333)` from clean `(0.013, 0.012, 0.008, 0.333)` HDR scene content. Those exploded HDR values propagate through the resolve → shared-memory → texture_load → gamma → composite chain, visible at the swapchain as high-frequency red/green chromatic speckle on Murphy and other surfaces.

## How it was localized

RenderDoc Pixel History on a noisy pixel in the lift scene — corruption point is **EID 9506** in a representative capture. A graphics-pipeline ownership transfer pixel shader (Pipeline State 1949) reads `xe_transfer_depth` + `xe_transfer_stencil` SRVs from a `kD24FS8 4xMSAA` source RT and writes to the `k_2_10_10_10_FLOAT 1xMSAA` HDR scene RT. Clean Xenia Canary avoids this entirely by keeping persistent host RTs and switching SRV views rather than running a format-converting transfer.

Step-by-step backward trace from the final swapchain noisy pixel:

1. **Final swapchain → noisy pixel.** Texture Viewer on `Cur Output 0` (the present-time swapchain image). Right-click a clearly noisy pixel on something that should be smoothly shaded (skin, fabric, wall). Pixel Context panel → **History**.
2. **Composite is the last writer.** The very last write to a swapchain pixel is a one-fragment `DrawInstanced(4, 1)` via `xe_bilinear_source` PS — a fullscreen quad that upscales an offscreen `R10G10B10A2_UNORM` 1280×720 LDR composite RT. The noise is already there in this source texture. Pipeline State → PS → Resources → note the SRV resource ID.
3. **Pixel History on the composite source.** Right-click same logical pixel position in the source texture → History. Single writer this time: a compute dispatch (`xe_apply_gamma_cs`).
4. **gamma_cs reads gamma_source.** Navigate to that dispatch's EID. SRVs are `xe_apply_gamma_ramp` (LUT) + `xe_apply_gamma_source` (the pre-gamma LDR scene RT, `R8G8B8A8_TYPELESS` viewed as UNorm). UAV is `xe_apply_gamma_dest`. Open `xe_apply_gamma_source` in Texture Viewer; the noise is there too.
5. **CopyTextureRegion + texture_load_cs + resolve_copy chain.** The single writer to the gamma source is a `CopyTextureRegion` from a staging buffer; the staging buffer was filled by `xe_texture_load_cs` (Compute Pipeline State 388) reading from `Buffer 317` (the 512 MB shared-memory buffer). The shared memory was populated earlier by a `Resolve Copy Fast 32bpp` dispatch reading from `EDRAM Buffer`. None of these *introduces* the noise — they faithfully propagate the corrupt bits from the host RT through the round-trip.
6. **Find the host RT and walk Colour Passes forward.** Event Browser → filter by "Dispatch" → look for the EDRAM dump that wrote the host RT contents to the EDRAM buffer (`RT Dump k_..._cs` followed by `Dispatch(N, M, 1)`). Trace the source of that dump back to its host RT — usually `RT @ 720t, <16t>, 1xMSAA, k_2_10_10_10_FLOAT` (the HDR scene buffer). Open the host RT in Texture Viewer. Walk forward through Colour Pass #1, #2, #3, ... selecting the OMSetRenderTargets event at each pass — the bound RT view shows the host RT state right before that pass starts. The Murphy/scene content stays **clean** through the actual game-draw Colour Passes (typically #1..#12). One specific event between Colour Pass M (clean) and Colour Pass M+K (broken) flips the host RT contents from clean HDR to exploded HDR.
7. **Pinpoint the divergent EID.** That one event is **always a `DrawInstanced(6, 1)` or `DrawIndexedInstanced(6, 1)`** with a graphics pipeline whose Pipeline State PS Resources list has SRVs named `xe_transfer_depth` and `xe_transfer_stencil`. These are the rexglue-sdk ownership-transfer shaders generated by `GetOrCreateTransferPipelines` in `src/graphics/d3d12/render_target_cache.cpp`.
8. **Read the corruption pattern off Pixel History.** Run Pixel History one more time, on the host RT pixel, scoped to just the divergent EID. The row will show `Tex Before` = clean HDR LDR values (~0.01 RGB), `Tex After` = explosive values such as `R = 8.1875, G = 0.25, B = 0.8125, A = 0.33325`. The `8.1875` is the smoking gun — it is the natural 7e3 range (`[0..31.875]`) showing depth bits being decoded as a 10-bit 7e3 float by `Float7e3To32` (lines ~3547-3553 of `render_target_cache.cpp` in the SDK).

## The fix

A narrow cvar in the ReXGlue SDK common render target cache — `skip_depth_color_7e3_aliasing_transfers` — drops the queued ownership transfer **only in the depth → 7e3 direction**. The reverse direction (7e3 → depth) is preserved because Downpour menu font rendering uses it legitimately; skipping that direction breaks the brightness slider screen (blue bands + ghost text).

Implementation lives in `RenderTargetCache::ChangeOwnership` inside `src/graphics/pipeline/render_target/cache.cpp`:

```cpp
bool skip_for_aliasing = false;
if (REXCVAR_GET(skip_depth_color_7e3_aliasing_transfers) &&
    transfer_source.is_depth && !dest.is_depth) {
  xenos::ColorRenderTargetFormat color_fmt = dest.GetColorFormat();
  skip_for_aliasing =
      color_fmt == xenos::ColorRenderTargetFormat::k_2_10_10_10_FLOAT ||
      color_fmt == xenos::ColorRenderTargetFormat::k_2_10_10_10_FLOAT_AS_16_16_16_16;
}

if (!transfer_source.IsEmpty() && transfer_source != dest && !skip_for_aliasing) {
  // existing transfer-queue code
}
```

CRITICAL: the condition is `transfer_source.is_depth && !dest.is_depth` (depth source → color dest only). The earlier symmetric variant (`is_depth != dest.is_depth`) catches both directions and breaks Downpour menu font rendering (blue bands + ghost text on brightness slider) because legitimate fonts use 7e3 → depth reinterpretation. **Always test menu after enabling the cvar**, not just the gameplay scene.

The cvar default is `false` so other titles are unaffected. The shipped `downpour.toml.sample` opts in.

## What NOT to do (dead ends from prior sessions, do not retry)

- `ChangeOwnership` is byte-identical to xenia — don't try to patch the common ownership logic.
- `OwnershipRange` / `AreOwnersSame` / `IsOwnedBy` are byte-identical — don't add provenance fields.
- DXGI format mapping functions (`GetColorResourceDXGIFormat`, `GetColorDrawDXGIFormat`, `GetColorOwnershipTransferDXGIFormat`) are byte-identical — don't touch.
- `host_depth_store` shader binding looks divergent from xenia (typed uint4 vs raw ByteAddressBuffer) — internally consistent in our SDK, **not** the bug. Tested via `dbg_disable_host_depth_alias` toggle.
- `TryResolveCopyDirectly` / `direct_host_resolve` cvar — rexglue-unique optimization but **not** the bug. Tested via `direct_host_resolve = false` forcing xenia-compat fallback path.
- Strict-roundtrip color-RT resurrection — works architecturally (1520 elides per scene) but does **not** change the visible bug. The strict-roundtrip path is innocent.
- Symmetric `skip_depth_color_7e3_aliasing_transfers` (both directions) — breaks menu, do NOT use; one direction only.
- ByteAddressBuffer migration of resolve_copy/host_depth_store family — large refactor, may not be necessary if the narrow cvar fixes the visible bug.
