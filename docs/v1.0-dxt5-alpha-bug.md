# DXT5 / BC3 alpha decode bug — EID 7117 forensic

A still-unfixed visual bug carried over into v1.0: UE3 SubUV smoke /
steam / fire / blood-decal particles render as **solid-black polygonal
billboards with sharp rectangular edges** instead of the soft
translucent sprites the artist authored. Most visible in the asylum
kitchen scene (Murphy in RS-273A uniform, facing the stove exhaust),
but the same decoder is used by every alpha-mask particle effect in
the game.

This document is the partially-completed forensic record. The work
needed to ship a fix is documented at the end.

## Symptom

* SubUV smoke / steam / fire particles render as opaque black quads.
* The INTERIOR of each sprite shows the faint correct grey smoke RGB.
* The alpha-feathered EDGES are binary opaque — no intermediate
  transparency, no soft falloff.
* Result: visible rectangular billboard edges around every particle.

## Root cause hypothesis

Bug in the SDK's X360 DXT5/BC3 alpha-block decoder. BC3 block layout:

```
2 bytes : alpha endpoints (alpha0, alpha1)
6 bytes : 16 × 3-bit interpolation indices (one per texel)
8 bytes : DXT1-style colour block
```

Endpoint decoding is correct (we see the right grey RGB inside the
sprite). The **3-bit index extraction is broken** — output alpha
snaps to exactly 0.0 OR 1.0 with no intermediate `1/7..6/7` values.

Suspect categories (in order of likelihood):

1. **3-bit endian / bit-order**: X360 stored its texture data
   big-endian or with reversed bit-order; the decoder may be reading
   the indices from the wrong half of each byte.
2. **X360 tile / swizzle byte position**: the X360 GPU swizzled
   texture data on upload. If the decoder is reading from the
   pre-swizzle byte offset it'll see endpoint bytes where indices
   should be.
3. **BC3 mode-7 vs dual-mode** (`alpha0 > alpha1` selector): UE3 may
   be using mode 7 (6-step interpolation + 0.0 + 1.0) and the decoder
   may be hardcoded to dual-mode (8-step interpolation).
4. **Interpolation formula**: even with correct indices, the
   `(7-i)/7 * a0 + i/7 * a1` step might be applied wrong.

PS3 (RPCS3) renders the same content correctly. PS3 RSX stores
textures natively in a different format and uses a different decode
path, so the bug is unique to the X360→D3D12 pipeline.

## RenderDoc forensic — what we did

Capture file `downpour_2026.06.23_16.03.28_frame2547.rdc` (asylum
kitchen scene). Already located the divergent EID; need to walk it
back to the texture data on disk.

### Step 1 — locate the broken pixel

Texture Viewer on `Cur Output 0` (swapchain), right-click a clearly
black pixel inside what should be a faint grey smoke wisp. Pixel
position recorded: **(743, 254)**.

### Step 2 — pixel history back to the source RT

`History` panel: the broken pixel originates from a `DrawInstanced`
batch chain. The relevant draws are:

| EID    | Pipeline state | Notes                                          |
|--------|----------------|------------------------------------------------|
| 7107   | UE3 SubUV PS   | `DrawInstanced(160, 1)` — first sprite batch  |
| **7117** | UE3 SubUV PS | `DrawInstanced(160, 1)` — second sprite batch |
| 7180   | RT transfer    | Innocent — samples already-broken RT          |
| 8036   | FSR3 RCAS      | Innocent — final upscale                      |

At EID 8036 the pixel shader output is
`R = 0.00293, G = 0.00293, B = 0.00293, A = 1.0` — almost-black RGB
(correct in absolute terms — light grey scaled down by tonemap) but
**alpha clamped to 1.0**. Should be `A ≈ 0.05-0.3` for proper alpha
blending of the sprite over the background.

### Step 3 — identify the bound SRVs

At EID 7117, the pixel shader has multiple `xe_textures_2d` SRVs
bound. The two primary suspects:

| SRV slot | Texture description                |
|----------|------------------------------------|
| **PS 0 [417]**   | "Wispy white/grey cloud" — DXT5 alpha mask |
| **PS 0 [767]**   | "Smoke puff round white-grey" — DXT5 alpha mask |

Both DXT5. Both used as `alpha-only` masks (RGB is constant from the
particle material; alpha is the per-pixel mask).

### Step 4 — Colour Pass output RT

Colour Pass #7 output target:
* 2560 × 4096 (super-sampled internal RT)
* 1× MSAA
* `R8G8B8A8_UNORM` (matches EDRAM 8-bit format)

The transfer at EID 7180 hits `k_2_10_10_10_10` HDR composite — still
within the SDK ownership-transfer machinery, NOT introducing the
alpha clamp. The clamp is already present in the bound texture data
at EID 7117.

## What's NOT the cause (confirmed)

We narrowed the bug down by changing other settings — all of these
have NO effect on the black-sprite artifact:

* `native_2x_msaa = true | false`
* `pso_missing_policy = skip | block | sync`
* Every `[SystemSettings]` particle / decal setting we toggled
* Resolution scale (1 / 2 / 3 / 4)
* `swap_post_effect = fxaa | mlaa | off`

This narrows the suspect surface to **purely** the DXT5 alpha-channel
decoder in `rexglue-sdk/src/graphics/texture/texture_cache.cpp` and
its sister `conversion.cpp`.

## What's left to do

**Estimated effort: 1 day.**

1. Open the .rdc capture at EID 7117 → texture viewer on
   `xe_textures_2d[417]` alpha channel. Confirm binary 0/1 transitions
   in the displayed image (the smoking gun — should show smooth alpha
   gradients).
2. Open `rexglue-sdk/src/graphics/texture/texture_cache.cpp` +
   `conversion.cpp`. Find the BC3 / DXT5 case.
3. Compare the alpha-index unpack
   `(block[2..7] >> shift) & 7` against Xenia's known-good
   `src/xenia/gpu/texture_util.cc ConvertBC3Block`. The bug is almost
   certainly a 1-line case-statement (e.g. dispatching to a DXT3
   decoder which has no per-texel alpha interpolation — that'd
   explain the binary 0/1 output).
4. Add a unit test that decodes a known DXT5 block against a
   reference output, so the fix doesn't regress on the next SDK
   merge.

## Severity

**High** for visual quality — affects every alpha-mask particle in
the game:

* Steam vents (asylum kitchen, sewer, plant)
* Cigarette smoke (Murphy's smoke break cutscene)
* Fire FX (lantern flames, burning vehicles)
* Fog volumes (street ambience)
* Blood decals (after combat, after enemy hits)

Low for gameplay — doesn't crash, doesn't affect controls.

## Why v1.0 ships with it

We had this fully scoped 2026-06-23 (RenderDoc capture, EID
identification, root-cause hypothesis). The 1-day fix budget didn't
fit into the v1.0 freeze window — every day in late June was being
spent on the per-frame stutter (PSO no-block-at-submission-end fix),
memexport readback work, and the launcher polish. Deferred to v1.0.1
with a clear plan above.
