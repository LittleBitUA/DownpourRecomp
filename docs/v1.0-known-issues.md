# v1.0 known issues + open follow-ups

What's *not* fixed in v1.0, why it wasn't fixed in this cycle, and where the
work continues. This document is the canonical "if you see X, here's why and
when it'll get addressed" list.

---

## 1. Depth-of-field "ефект скла" haze still present

**Symptom**: indoor cutscenes and certain camera framings still apply the
X360 build's depth-of-field blur, giving distant geometry a soft "glass"
look. Most visible at scene transitions and in tight interiors.

**Why it wasn't fixed in v1.0**:

We tried two routes, both blocked:

1. **Coalesced.ini override.** Added `[Engine.DOFEffect]
   BlurKernelSize=0.0`, `[Engine.UberPostProcessEffect] BlurKernelSize=0.0`,
   `[OurEngine.OurUberPostProcessEffect] DOF_BlurKernelSize=0.0`, etc. into
   `Coalesced_INT.bin`. Tested 2026-06-27 — DoF unchanged on the Murphy
   intro bridge scene. Conclusion: `BlurKernelSize` is NOT a `config`
   property in Konami's UE3 fork, so .ini sections are silently ignored.
   Coalesced reverted.
2. **Engine.xxx byte patch.** Identified the 13 baked-CDO float offsets
   for `BlurKernelSize` / `DOF_BlurKernelSize` / `DOF_BlurBloomKernelSize`
   across `Engine.xxx` and `OurEngine.xxx`. Wrote a Python tool that
   decompresses (via Gildor's `decompress.exe`), patches the floats to
   `0.0`, and replaces the live files. Game hung on infinite loading
   screen because the recompiled X360 UE3 loader expects the original
   LZX-wrapped cooked format, not the Gildor PC-format output.

**Path forward — v1.0.1 or later**:

Integrate an open-source X360 LZX encoder. Best candidate identified is
`elasota/liblzx` (https://github.com/elasota/liblzx) — pure C, CAB-LZX
encoder, closest open analogue to Microsoft's `xcompress32.dll` (which is
proprietary leaked-SDK and not redistributable). Realistic effort: 4-8
hours including:

* Build liblzx as a CLI `lzx_pack input.bin output.lzx`.
* Wrap each compressed chunk in the XMemCompress per-block header
  (4-byte BE compressed-size + uncompressed-size — documented in bnnm's
  LZX-XNB gist + Xenia's `src/xenia/cpu/lzx.cc` decoder).
* Round-trip-validate against Gildor's `decompress.exe`.
* Wire into the existing `patch_blur_kernel.py` decompress → patch →
  recompress pipeline.

Fallback: `wimlib lzx-compress.c` if liblzx's window-size tuning
diverges from the Xbox stream.

**Workaround today**: The launcher's UE3 Engine tab exposes
`DepthOfField` as a `[SystemSettings]` toggle (it IS config). Setting it
to `False` disables the entire `UberPostProcess` chain — including DoF.
The trade-off is that it also disables bloom, motion blur, colour grade,
ambient occlusion, and anything else routed through `UberPost`. Most
players find this too aggressive; the targeted `BlurKernelSize=0` scalpel
is the v1.0.1 work item.

## 2. Foliage cyan / purple specks on certain mip transitions

**Symptom**: occasional pixels in foliage / mid-distance shrubs flicker
cyan or purple as the camera moves. Most visible in outdoor day scenes
with dense vegetation (Pearl Creek, motel exterior).

**Why it wasn't fixed**:

This is the same general class as the chromatic-noise pixel bug
documented in `docs/chromatic-noise-fix.md` — the X360's 7e3 HDR format
(`k_2_10_10_10_FLOAT` with 7-bit mantissa, 3-bit exponent per channel)
doesn't decode cleanly through the fallback `resolve_full_*_cs.h`
compute shaders that the SDK uses when ROV-side resolves are bypassed.
We confirmed the existing upstream Xenia shaders are bytecode-identical
to ours and exhibit the same flicker.

A clean fix requires a custom DXBC pass in `TryResolveCopyDirectly` that
performs a proper 7e3 → float16 conversion with the right saturation
clamps. Estimated effort: 1-3 days for someone familiar with DXBC.

**Workaround today**: None. The specks are far less visible at
`resolution_scale = 2` (default — they get averaged into the SSAA
downscale). At `resolution_scale = 1` they're more obvious.

**Reference state in v0.1.1 vs v1.0**: visually identical on the same
scene — this regression did NOT happen during the v1.0 cycle, it's been
present since the first port. We re-verified against stock Xenia Canary
2026-06-18 on the same TU1 build: identical. So it's not a v1.0
regression, just a still-unfixed inherited bug.

## 3. CRITICAL spikes during scene transitions / level loads

**Symptom**: very occasionally (~2 events per 10 minutes of play) a
single frame takes 250-360 ms. Visible as a brief freeze when transitioning
between Silent Hill streets / interiors, when loading a save, or when
heavy cutscene content arrives.

**Why it wasn't fixed**:

These spikes have a clear signature in the PSO_STALL log — 450-525
`fence_wait_submission` calls in one frame, totalling 120-160 ms. The
cause is Case A of the memexport readback path being called per-key
(37+ memexport keys per frame in the busiest scenes), each triggering an
`AwaitAllQueueOperationsCompletion` to ensure the GPU finished its
write before the CPU read.

The canonical fix is a batched end-of-frame `AwaitAll` — one fence wait
covers all per-key readbacks. We have memory of the design but two prior
attempts to short-circuit the per-key fence (memset zero-init the
destination, and plain early-return without writing anything) both TDR'd
the GPU because UE3's PPC code reads the destination memory and feeds it
into indirect-draw counts and skinning matrices. The CPU MUST receive
valid post-memexport data.

**Path forward — v1.0.1**:

Refactor `D3D12CommandProcessor::IssueDraw_MemexportReadbackFastPath` to
queue per-key readback work into a per-frame batch, then drain all batched
work with a single `AwaitAllQueueOperationsCompletion` at the end of the
frame (or at the next memexport-result consumer). Estimated effort: 2-3
hours of careful work plus several hours of TDR-testing.

**Player impact today**: 0.04% of frames at the test bench — visible only
as occasional sub-second hitch at scene boundary. Does not affect
mid-scene gameplay; user playtest confirmed feel is "like a native port".

## 4. Mid-session SEVERE spikes (50-93 ms) every ~30 seconds

**Symptom**: occasional brief frame-time spikes during steady gameplay
that don't correlate with any visible game event. Not jarring, but a
keen-eyed observer can notice them.

**Diagnostic finding**:

These show `fence_wait_ms = 0`, `submit_ms` and `present_ms` near zero —
the render thread is completely idle. The only signal is
`guest_cpu_wait_ms = 250-700 ms across 50-100 wait calls`. That's
multiple guest threads (UE3 audio mixer, background loader, render
proxy, game thread) all blocking briefly on shared synchronization
primitives at the same time.

**Why it wasn't fixed**:

This is UE3 internal thread synchronization behaviour — fixing it
would require changes to the original game's threading model, which is
baked into the recompiled binary. Not feasible without UE3 source.

**Path forward**:

Possibly mitigable by adjusting the guest scheduler / thread priority
mapping in the SDK's kernel emulation layer. Speculative. Not on any
roadmap.

**Player impact**: 0.13% of frames — practically invisible. Not on the
v1.0.1 fix list.

## 5. PSO library blob is GPU-vendor-specific

**Symptom**: first launch on a fresh GPU triggers a ~13 second cold-compile
of ~1700 ROV pixel shaders before the game window appears. Subsequent
launches are instant.

**Why this is "as designed"**:

D3D12 `ID3D12PipelineLibrary` stores the driver's compiled PSO microcode
in the format the GPU vendor's driver expects. The blob is NOT portable
across vendors (NVIDIA blob won't load on AMD), so we don't ship it
pre-baked in the zip. The user's first launch builds it locally to
`cache/shaders/local/<title>.<scale>.<path>.d3d12.pso_lib` (~234 MB on
RTX 5070).

**Workaround**: None needed — it only happens once per GPU install. If
you switch GPUs (or driver version), the blob is regenerated on next
launch.

**Path forward**: nothing planned. This is fundamentally how D3D12 PSO
libraries work.

## 6. Black screen + audio if game data is missing or wrong region

**Symptom**: launcher works, "Start Game" launches `downpour.exe`, but
the game window opens black and stays black. Audio may or may not play.

**Diagnosis**:

The recompiled binary expects USA or Europe X360 cook data
(title id `4B4E0823`, base XEX hash `7A3D5809776EE6AB`). Other regions
(Japan, China, demo / preview builds) have different XEX hashes and
different cook-data layout — the loader gets confused but doesn't crash
hard, just produces no usable output.

**Workaround**: confirm your X360 game files are USA or Europe. If
you're not sure, hash `assets/default.xex` — should be
`7A3D5809776EE6AB`.

**Path forward**: support for additional regions would require a fresh
codegen pass per region (different XEX → different recompiled C++) and
a launcher-side switch to pick the right binary. Not on any roadmap.

## 7. Other UE3 quirks inherited from the X360 build

These are present in the X360 release and would be in scope only for a
hypothetical major engine-side rework. v1.0 doesn't touch any of them:

* **No native HDR**, no ray tracing, no DLSS frame generation. (UE3 was
  pre-HDR.)
* **No aspect-correct ultrawide** — 21:9 monitors get a stretched 16:9
  image. Fixing this requires a UE3 FOV / projection patch (similar to
  Skate 3 Recomp's approach).
* **Resolution is internally clamped** — internal render targets above
  ~4K start trashing the EDRAM emulation buffers. `resolution_scale = 4`
  technically works but isn't recommended above `2` on RTX 5070-class
  GPUs.
* **No mid-game save**. The X360 build's save system is checkpoint-only.

These are 2012 design limits, not v1.0 regressions.

## Reporting an issue not covered here

If you hit something not in this list, please open a GitHub issue with:

1. The launcher version (visible in window title — should be `v1.0`).
2. A `logs/downpour_NNN.log` capture taken with `log_to_file = on` +
   `log_level = warn` in the Debug tab.
3. Steps to reproduce, including the scene / save you were in.
4. GPU model + driver version (the early log lines record this).

A short video / screenshot helps a lot for visual bugs.
