# Resolution scale notes — the three distinct issues

The `resolution_scale` cvar (and its X / Y twins) controls internal
super-sampling. The history of this knob is messier than a single
"higher is better" rule. There are actually **three distinct issues**
tangled together, two of which are real and one of which turned out
to be a phantom caused by a different cvar entirely.

This document untangles them so you know what's safe to change.

## Issue 3a — HUD text garbling (PHANTOM — was actually memexport)

**What looked like the problem (corrected 2026-06-20)**:

Earlier in the project, raising `resolution_scale` from 1 to 2
appeared to garble UE3 fonts — subtitles, menu text, ammo counts —
into unreadable smudges. Conventional wisdom on the project was
"scale=2 breaks HUD text".

**What was actually causing it**:

`readback_memexport = false` (a Skate3-fork default we briefly
imported into the Downpour SDK config) **alone** garbled the font
atlas. The bug had nothing to do with super-sampling. Removing only
that one setting restored clean text at scale=2.

**Why we thought it was scale**:

The Skate3-fork imported many cvars at once. We toggled scale and
memexport in the same session and incorrectly attributed the
garbling to scale.

**Hard rule for v1.0 and beyond**:

* `readback_memexport = true` (the SDK default). Do NOT flip to false
  for Downpour — it WILL garble UE3 fonts at any resolution scale.

Skate3-derived ports inherit `readback_memexport = false` because
their HUD pipeline doesn't depend on memexport. Downpour's does.

## Issue 3b — SSAO precision bug worsens with fractional scale

**Symptom**: ordered-dither / screen-door pattern on shadowed regions
(Murphy in shadow + dark cave walls during cutscenes). Lighting-
correlated — present only in shadows, **NOT** on lit foreground.

**Root cause**: SSAO half-res precision + broken bilinear upsample
inside UE3's post-process chain. The ambient occlusion buffer is
computed at half resolution (a 2012 X360 optimization) and upsampled
back with bilinear filtering. The dither pattern is supposed to be
hidden by the upsample blur. When the final present chain ALSO
applies a downscale (because internal RT > output res), the dither
pattern survives both passes and aliases against the output pixel
grid.

**A/B verified 2026-06-20** on the same scene:

| Scale | Output downsample | SSAO dither pattern             |
|-------|-------------------|---------------------------------|
| 1     | 720p → 1080p **upscale** (bilinear blur) | **Softer** — upscale-blur low-passes the dither for free |
| 2     | 1440p → 1080p, factor 1.33× | **Worse** — fractional downsample misaligns output grid with source dither |
| 3     | 2160p → 1080p, **integer 2×** | **Best dither suppression** — clean 2× box filter averages the half-res SSAO cleanly |
| 4     | 2880p → 1080p, factor 2.67× | Heavy but visually similar to scale=3, with much more GPU cost |

Same dither artifact reproduces in **stock Xenia Canary** on the same
TU1 build — so it's an upstream Xenos→D3D12 issue, not a rexglue
regression. Not something we can fix without modifying the SSAO
upsample shader.

**Recommendation**: scale=1 if your GPU can't comfortably handle SSAA
(SSAO dither hidden for free). Scale=3 if you want sharpest output
and can afford the cost. Scale=2 is the canonical v1.0 default
because FXAA (also on by default) does some of the dither-suppression
work that scale=3 would do — at much lower GPU cost.

## Issue 3c — UE3 PostProcess RTs are stuck at 720p (architectural ceiling)

**This is the most important resolution-related finding in v1.0.**

Diagnosed 2026-06-22 late night via SDK log line at
`command_processor.cpp:2318`:

```
"D3D12 draw resolution scaling is enabled, but the swap source is unscaled..."
```

**What this means in practice**:

The EDRAM (in the X360, the dedicated render-target memory) is
emulated at the **scaled** resolution. At scale=2 the SDK renders
into a 2560×1440 EDRAM-equivalent buffer; at scale=3, 3840×2160.

BUT — UE3's cooked content includes post-process render targets
allocated at the guest-native 1280×720. The post-FX chain
(depth-of-field, bloom, tonemap, colour grading) reads from
**those 720p targets, not the EDRAM**. The SDK has no way to know
about game-controlled texture allocations, so it can't scale them.

Pipeline summary:

```
   ┌─ EDRAM (scaled)  ─────  high-res draws happen here  ─┐
   │                                                      │
   │  ↓ UE3 reads from EDRAM into its own 720p RTs        │
   │                                                      │
   ├─ UE3 PostProcess RTs (720p, immutable from outside) ─┤
   │                                                      │
   │  ↓ DoF / Bloom / Tonemap happens here at 720p        │
   │                                                      │
   ├─ Final composite (720p)                              │
   │                                                      │
   │  ↓ SDK present chain upscales 720p → output res      │
   │                                                      │
   └─ Swapchain (1080p / 1440p / 4K)                      ─┘
```

**Practical consequence**: all the super-sampling work done in EDRAM
gets **averaged-down by UE3's post-process** before it reaches the
swapchain. Beyond a certain scale, you're paying the GPU cost without
any visible benefit.

**Workaround / mitigation**:

* `swap_post_effect = fxaa` masks the differences between
  `present_effect = bilinear / box / cas`. FXAA smears the
  upscale-induced softness into something that looks "fine" on
  screen.
* Setting `bUseMaxQualityMode = True` (in the launcher's UE3 Engine
  tab, under SystemSettings) MAY force UE3's PostProcess RTs to
  full-resolution. This works because Konami evidently added the
  `config` keyword to that property in their fork. Visual confirmation
  is mixed — some scenes look identical, others slightly sharper.
  Off by default in v1.0; users can experiment.

**The real fix** would be an SDK-level patch to UE3's RT allocation
that scales the PostProcess targets in sync with EDRAM. That's a
non-trivial engine-side hook (per-target classification, format
matching) and is deferred indefinitely.

## Issue 3d — Foliage cyan/purple specks (unrelated to scale)

Listed here for completeness because it sometimes gets blamed on
resolution scale. It's actually independent.

* The cyan/purple flicker on outdoor foliage is the 7e3→float16
  resolve fallback bug, baked into the shared rexglue + Xenia
  codegen.
* Identical in stock Xenia Canary on the same TU1 build.
* Mitigated (NOT fixed) by `swap_post_effect = fxaa`.
* Independent of `resolution_scale` — the artifact shows at all
  scale values.

See `v1.0-known-issues.md` for the cleanup status.

## Recommended settings

| You want… | Recommended values                                  |
|-----------|-----------------------------------------------------|
| Best perf, soft picture | `resolution_scale = 1`, `swap_post_effect = fxaa`, `present_effect = fsr3` |
| Sharper picture, fxaa-blurred dither | `resolution_scale = 2`, `swap_post_effect = fxaa`, `native_2x_msaa = true` (v1.0 default) |
| Sharpest, no fxaa smear  | `resolution_scale = 3`, `swap_post_effect = off` (4K-class internal RT, heavy GPU cost) |
| Anything else            | Don't touch `readback_memexport` — keep at `true` |

## Hard rules (don't break these)

* `readback_memexport = true` — flipping to false WILL garble UE3
  fonts regardless of resolution scale. The Skate3-fork ports get
  away with `false` because their HUD pipeline doesn't depend on
  memexport; Downpour's does.
* `gpu_allow_invalid_fetch_constants = false` — SDK 0.8.1.19+ flipped
  this to true upstream. Downpour MUST hard-pin to false; the launcher
  EnsurePerfDefaultsInToml seeds it.
* `execute_unclipped_draw_vs_on_cpu = false` — same situation,
  same hard pin.

These three correctness pins together cost about 15-20% of GPU
budget vs the "fast but wrong" settings. The cost is well worth it —
without them, the game looks WORSE than the X360 original.
